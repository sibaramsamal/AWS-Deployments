## CI-CD implementation Using Github Actions

🔐 PART A — [Cloning the repositiry](../Clone%20Repo/0.%20Generation%20of%20SSH%20Key.md)  
PART B — GitHub Actions → EC2 (for auto deploy)   
Goal: Allow GitHub servers to SSH into your EC2.  

Steps:
1. Create CI key secret
2. Add public key to EC2
3. Add in the github secret and keys
4. update workflow rules in .github/workflows/
4. ceate .sh files inside server where code files are there
5. give permission for executable
6. test by pushing to dev

Overview:
```
SSH works like this:
GitHub → sends PRIVATE key
EC2 → checks PUBLIC key in authorized_keys

👉 Names don’t matter
👉 Key pair matching matters
```

### 1. Generate CI SSH key (on EC2 or local machine)    
`ssh-keygen -t ed25519 -f <any name of your choice> -N "" -C "Any valid comment"`
```
-t ed25519 (don’t change casually).   
It defines key type.  
ed25519 = modern, secure, fast (recommended)

👉 You can use rsa, but don’t unless you have a reason.

⚠️ -N "" → IMPORTANT (not exactly static, but critical)
-N = passphrase
"" = no password

✅ -C "github-actions" → DYNAMIC (just a label)
This is just a comment inside the key
Helps identify the key later

⚠️ Permissions → STRICT 
chmod 700 ~/.ssh
chmod 600 ~/.ssh/authorized_keys
```

Let's create a key: `ssh-keygen -t ed25519 -f ~/.ssh/hrms-ci-deploy -N "" -C "hrms-ci-deploy"`

### 🔐 2. Add PUBLIC key to EC2:
`cat ~/.ssh/hrms-ci-deploy.pub >> ~/.ssh/authorized_keys`
```
By adding the key to ~/.ssh/authorized_keys file, we are making sure that EC2 will communicate when github actions trigger.

From the key generation in the above process, the resulting files are:
~/.ssh/hrms-ci-deploy
~/.ssh/hrms-ci-deploy.pub

Then fix permissions of the files(if using first time or manually cerated):
chmod 700 ~/.ssh
chmod 600 ~/.ssh/authorized_keys

⚠️ To verify whether the key added in to EC2 or not, simply open the "authorized_keys" file. You will see one extra line added to it. This is the public key which we just created.

👉 This allows anyone with the private key to SSH into this EC2.
```

### 3. Copy PRIVATE key to GitHub
`cat ~/.ssh/hrms-ci-deploy`

```
👉 Copy the entire output (including -----BEGIN OPENSSH PRIVATE KEY-----)  

Go to your repo:
Settings → Secrets and variables → Actions → New repository secret. Add the values.

eg.
Name: HRMS_CI_DEPLOY_ACTION_KEY
Value: (paste private key you copied)
```

### 4. Create GitHub Actions workflow
- Create file: `.github/workflows/deploy.yml` in your project.

minimal working setup:
```
name: Auto Deploy Backend

on:
  push:
    branches:
      - dev
      - main

jobs:
  deploy:
    runs-on: ubuntu-latest

    steps:
      - name: Deploy to EC2
        uses: appleboy/ssh-action@v1.0.3
        with:
          host: 13.126.227.130 ## this one also we can get from ${{ secrets.HOST }}
          username: ubuntu
          key: ${{ secrets.HRMS_CI_DEPLOY_ACTION_KEY }}
          script: |
            if [ "${{ github.ref_name }}" = "dev" ]; then
              cd /opt/apps/DQ-HRMS-BE-DEV
              ./deploy-dev.sh
            elif [ "${{ github.ref_name }}" = "main" ]; then
              cd /opt/apps/DQ-HRMS-BE-PROD
              ./deploy-prod.sh
            fi 
```

Never create deployment CI/CD script file using sudo

⚠️ Always use simply `nano /opt/apps/backend/deploy-dev.sh`

Then paste your script there.
- Dev environment script file: deploy-dev.sh
```
#!/bin/bash
set -e

echo "Deploying backend Development Environment..."

cd /opt/apps/DQ-HRMS-BE-DEV

git pull origin dev

./mvnw clean package -DskipTests

sudo systemctl restart dq-hrms-dev

echo "Backend of Development environment deployed successfully"
```

- Prod environment script file: deploy-prod.sh
```
#!/bin/bash
set -e

echo "Deploying backend Production Environment..."

cd /opt/apps/DQ-HRMS-BE-PROD

git pull origin main

./mvnw clean package -DskipTests

sudo systemctl restart dq-hrms-prod

echo "Backend of Production environment deployed successfully"

```

⚠️ Then assign permissions: `chmod +x deploy-backend-main.sh`

We can place this deployment script logic inside workflows directly(inline deployment logic) but this approach(script-based deployment) is generally better because in case of any issue in deplymnet happens, we can easily debug it using the script files. eg. `./deploy-frontend-dev.sh`
